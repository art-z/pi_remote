# Сервис `audio`: микрофон → Vosk (русский) → дисплей через Redis

Агент читает **сырой PCM** с ALSA (`arecord`), распознаёт речь моделью **Vosk small-ru** и обновляет тот же ключ Redis, что и веб-интерфейс: `display:state`, плюс публикация в `display:notify` для мгновенной перерисовки на ST7789.

Сборка образа — **на Raspberry Pi** (или `docker buildx` под `linux/arm64`), см. корневой `README.md`.

## Модель Vosk (на диске, не в образе)

Образ **не** качает модель: каталог с распакованной **small-ru** монтируется read-only в `/opt/vosk-model`.

- **Прямая ссылка на ZIP:** https://alphacephei.com/vosk/models/vosk-model-small-ru-0.22.zip  
- Куда положить и как распаковать: **`models/README.md`** в корне репозитория.  
- Другой путь на хосте: переменная **`VOSK_MODEL_HOST_DIR`** в `.env` (см. `docker-compose.yml` у сервиса `audio`).

Внутри контейнера путь к модели задаётся **`AUDIO_MODEL_PATH`** (по умолчанию `/opt/vosk-model` и должен совпадать с правой частью volume).

## Устройство захвата: `AUDIO_ALSA_DEVICE`

В `.env` задаётся строка **имени PCM-устройства ALSA**, которое передаётся в `arecord -D …`.

### Что значит `plughw:1,0`

ALSA адресует карты и устройства числами:

- **Первая цифра** — **номер карты** (card), как в выводе `arecord -l`.
- **Вторая** — **номер устройства** (device) для **захвата** (обычно `0`, если одно устройство ввода).

Префиксы:

| Префикс   | Смысл |
|-----------|--------|
| **`hw:`** | «Железо как есть»: формат и частота должны совпасть с возможностями устройства. Для 16 kHz mono часто подходит, но не всегда. |
| **`plughw:`** | Тот же card/device, но через **plug** — ALSA при необходимости **конвертирует** формат/частоту/каналы. Для контейнера и USB-микрофонов **чаще всего надёжнее начать с `plughw:`**. |

Пример: карта **1**, устройство захвата **0** → **`plughw:1,0`**.

Имя **`default`** — логическое устройство по умолчанию (часто `pulse` на десктопе). На минимальной Ubuntu на Pi внутри Docker оно иногда **не указывает на нужный микрофон** или ведёт себя иначе, чем на хосте. Если тишина или ошибка `arecord` — явно укажите `plughw:CARD,DEVICE` по списку ниже.

## Как найти правильное устройство

### 1. На **хосте** (малина), не в контейнере

```bash
arecord -l
```

Ищите строки вида `card 1: USB …` и `device 0: USB Audio …` — для захвата обычно смотрят на **device** под нужной **card**. Тогда в `.env`:

```env
AUDIO_ALSA_DEVICE=plughw:1,0
```

(подставьте свои номера вместо `1` и `0`).

Полный список **имён** (в т.ч. удобные строки вроде `hw:CARD=Device,DEV=0`):

```bash
arecord -L
```

Можно использовать и такие имена, если они стабильны на вашей системе.

### 2. Проверка, что микрофон реально пишет

На хосте (подставьте своё `-D`):

```bash
arecord -D plughw:1,0 -f S16_LE -c 1 -r 16000 -d 3 -t wav /tmp/test.wav
aplay /tmp/test.wav
```

Если файл не растёт или ошибка — номер карты/устройства неверный, нет прав, или устройство занято другим процессом.

### 3. В Docker

Контейнер получает **`/dev/snd`** (см. `docker-compose.yml`). **Номера карт внутри контейнера обычно совпадают с хостом**, но если ставили `pulseaudio` / меняли конфиг — перепроверьте `arecord -l` **внутри** контейнера:

```bash
docker compose exec audio arecord -l
```

Затем пропишите то же значение `AUDIO_ALSA_DEVICE` в `.env` на хосте и перезапустите сервис:

```bash
docker compose up -d audio
```

Логи:

```bash
docker compose logs -f audio
```

Сообщения о завершении `arecord` или ALSA чаще всего означают неверный `-D` или отсутствие микрофона.

### `libvosk.so` / `libatomic.so.1: No such file or directory`

В образе `audio` должен быть пакет **`libatomic1`** (зависимость нативной библиотеки из wheel Vosk). Если собирали старый Dockerfile без него — пересоберите образ: `docker compose build --no-cache audio`.

## Остальные переменные (кратко)

| Переменная | По умолчанию | Назначение |
|------------|----------------|------------|
| `REDIS_URL` | `redis://redis:6379/0` | Подключение к Redis (как у `api` / `display`). |
| `DISPLAY_STATE_KEY` / `DISPLAY_NOTIFY_CHANNEL` | как в корневом `.env.example` | Куда писать текст и сигнал дисплею. |
| `AUDIO_MODEL_PATH` | `/opt/vosk-model` | Каталог модели **внутри контейнера**; совпадает с mount из `VOSK_MODEL_HOST_DIR`. Другая модель — другой mount + при необходимости другой путь в `AUDIO_MODEL_PATH`. |
| `VOSK_MODEL_HOST_DIR` | `./models/vosk-model-small-ru-0.22` | Только в **docker-compose** / `.env`: откуда на хосте брать распакованную модель. |
| `AUDIO_SAMPLE_RATE` | `16000` | Частота захвата; под эту модель Vosk нужен **16 kHz**. |
| `AUDIO_CHUNK_BYTES` | `8000` | Размер чанка чтения из `arecord` (кратно 2 байтам на сэмпл). |
| `AUDIO_PARTIAL_PUBLISH_SEC` | `0.25` | Не чаще этого интервала слать **частичный** текст в Redis (меньше мигания на экране). |
| `AUDIO_DISPLAY_MODE` | `pulse` | `pulse` или `status` — как отображать на дисплее (см. `viz.py` / `display_agent.py`). |

Поведение дисплея (`rotate`, `font_size`) агент **не затирает**: он мержит JSON с уже сохранённым состоянием.

---

## Без микрофона или без ALSA на хосте

Закомментируйте сервис **`audio`** в корневом `docker-compose.yml`, чтобы стек не перезапускал падающий контейнер.
